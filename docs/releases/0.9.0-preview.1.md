# BotFramework 0.9.0-preview.1

`0.9.0-preview.1` is the first intentionally breaking preview of the public
BotFramework package surface. It is a preview for framework consumers, not a
promise of source compatibility with the repository's legacy demonstration
games.

## Highlights

- Tenant-aware public contracts with opaque `TenantId`, `ScopeId`, `PlayerId`
  and `RequestId` values.
- Canonical REST routes under
  `/api/v1/tenants/{tenantId}/scopes/{scopeId}/...` with idempotency headers,
  correlation headers and stable RFC 7807 error codes.
- Typed `BotFramework.Client` REST transport and checked-in NSwag client
  generation.
- Redis-backed multi-dimensional rate limiting with bounded local fallback and
  recovery after Redis becomes available again.
- OpenTelemetry Collector, Tempo, Prometheus, Grafana trace links and
  Alertmanager deployment defaults.
- The package-only CoinFlip consumer sample and
  `BotFramework.Templates::0.9.0-preview.1` `dotnet new botframework-game`
  scaffold.
- `BotFramework.Host.Workflows` durable workflow infrastructure for long-lived
  operations: PostgreSQL step persistence, Wolverine durable delivery,
  correlation, partitioning, retries, replay and generic saga state.
- Meta tournament workflow integration with stable wallet operation ids,
  explicit rollback for definitive rejection and `Pending` responses while a
  transiently failed remote operation is retried.

## Persistence and service boundary

The durable workflow table is owned by the process that registers the workflow
and uses that process's `ConnectionStrings:Postgres`. In the microservice
topology this is the Backend database, not the Wallet or Identity database.
Workflow persistence is an operator-visible recovery timeline; domain truth
still belongs to the module's own transaction and state tables.

Cross-service effects are coordinated with contracts, outbox delivery,
idempotent operation ids and explicit rollback operations. This release does
not introduce a distributed PostgreSQL transaction.

## Published package set

The package-only release contains:

| Package | Purpose |
|---|---|
| `BotFramework.Contracts` | Tenant, transport, error, pagination and limiter contracts |
| `BotFramework.Sdk` | Module, domain, decision/effect and scheduling abstractions |
| `BotFramework.Testing` | Test fakes and in-memory fixtures |
| `BotFramework.Rest` | Tenant-aware REST middleware and module route contract |
| `BotFramework.Telegram.Abstractions` | Telegram update and tenant-resolution contracts |
| `BotFramework.Discord.Abstractions` | Discord tenant-resolution contracts |
| `BotFramework.Client` | Typed REST client and generated transport |
| `BotFramework.Scheduling.Abstractions` | Transport-neutral scheduled command contracts |
| `BotFramework.Templates` | `dotnet new botframework-game` project template |

`BotFramework.Host`, `BotFramework.Telegram`, `BotFramework.Discord`,
`BotFramework.Scheduling.Quartz` and `BotFramework.Rendering` remain runtime
composition projects in this preview. They are built and tested in the
repository but are not published as public package dependencies. In
particular, the durable workflow implementation is currently consumed by
repository composition roots and the CoinFlip workflow example through a
source reference.

## Upgrade notes

This is a breaking preview. Consumers must migrate to tenant-aware routes and
contracts, opaque ids, string idempotency keys and the new error envelope.
Do not use `BotFramework.Host.Workflows` as a replacement for a local atomic
domain commit: a workflow records progress and retries remote work, while the
module's AtomicEffect/transaction boundary remains the source of truth for its
own database.
